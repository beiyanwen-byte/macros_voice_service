## 🔧 关键技术决策

### 1️⃣ 版本兼容性选择（最重要！）

**问题背景**:
- FunASR 最新版 (1.3.x) 要求 PyTorch 2.4+
- macOS CPU 最高仅支持 PyTorch 2.2.2
- 盲目升级会导致 "CUDA not available" 错误

**解决方案**:
```txt
锁定版本组合:
- funasr==1.1.9
- torch==2.2.2
- numpy==1.26.4 (<2.0, 避免兼容性问题)
```

**经验教训**:
❌ **错误流程**: 直接使用最新版 → 遇到兼容性错误 → 花 2 天调试  
✅ **正确流程**: 提前查阅官方文档 → 在目标环境做最小化测试 → 锁定已知可用版本

**通用原则**:
> 任何大模型本地部署项目，必须**第一步**检查硬件兼容性（CPU/GPU/操作系统），不要假设最新可用！

---

### 2️⃣ 懒加载模式设计

**用户需求**: "服务启动太慢，等待 3 分钟很焦虑"

**技术方案对比**:

| 方案 | 启动速度 | 首次识别速度 | 用户体验 |
|------|---------|------------|---------|
| 预加载模型 | ~3 分钟 | <1 秒 | ❌ "为什么这么久还没反应？" |
| **懒加载** | <1 秒 | ~3 分钟 | ✅ "秒开！点击开始后会提示需要等待" |

**实现逻辑**:
```python
# 启动时不加载模型
def start_asr():
    # 用户点击开始时才启动子进程
    subprocess.Popen([python, "asr_core.py", ...])
    
# asr_core.py 内部延迟加载
def main():
    print("正在加载模型...")  # 此时才真正加载
    model = AutoModel(model="paraformer-zh", device="cpu")
```

**适用场景扩展**:
- LLM 本地推理服务
- 数据库连接池预热
- 任何初始化耗时 > 1 秒的场景

### 3️⃣ 滑动窗口参数优化

**问题现象**: 初始使用 0.5s 窗口导致输出碎片化
```
原始输出：
[14:50:35] 这个 不同 的 这个 呃
[14:50:36] 反思 甚至 批判 啊 就是 我们
```

**优化过程**:
| 窗口大小 | 语义完整性 | 实时性 | 最终选择 |
|---------|----------|--------|---------|
| 0.5s | ❌ 极度碎片 | ⭐⭐⭐⭐⭐ | 放弃 |
| 3s | ⚠️ 部分完整 | ⭐⭐⭐⭐ | 备选 |
| **10s** | ✅ 句子级完整 | ⭐⭐⭐ | **选中** |

**权衡思考**:
> 牺牲一定的实时性（从 0.5s → 10s），换取更好的可读性和后续编辑便利性。对于学习记录、会议纪要等场景，完整度比极速更重要。

---

### 4️⃣ 静音过滤机制

**问题**: 系统无声时会输出乱码或空白识别结果

**解决方案**: RMS 音量阈值检测
```python
def calculate_rms(audio_data):
    """计算音频能量"""
    return sqrt(mean(audio_data ** 2))

if rms < 0.04:
    skip_recognition()  # 跳过本次识别
```

**效果**:
- 播放无声视频 → 无垃圾输出 ✅
- 正常说话 → 准确识别 ✅
- 可调节参数 `volume_threshold=0.04` 以适应不同环境

## 💡 重要经验沉淀

### A. 资源占用透明化

**用户疑问**: "程序会不会产生大量缓存占用硬盘？"

**改进前的文档缺陷**:
- README 只说"系统内录"和"实时显示"
- 没有提及内存/CPU/磁盘的具体用量
- 用户只能猜测或试错

**改进后的做法**:

在 README 增加独立章节："💾 资源占用预估（使用前必读）"

```markdown
启动前：
- 磁盘空间：需要 2GB 空闲（模型下载）
- 内存：至少 2GB 空闲（模型加载）

运行时：
- CPU: 识别时 ~75%，空闲 <5%
- 内存：常驻 ~800MB
- 网络：首次加载需下载模型，后续离线可用

停止后：
- 仅保留 transcripts/*.txt（每次约 5KB/分钟音频）
- 不会生成临时文件或日志堆积
```

**关键洞察**:
> 技术文档不仅要说明"怎么用"，还要告诉用户"会发生什么"。资源占用是影响用户决策的关键信息，应该在**启动前**就说明清楚。

### B. 交互细节决定体验

**本次做到的亮点**:
1. ✅ 双 Tab 切换：加载中看日志 → 完成看转录内容
2. ✅ 累计时间实时更新：让用户知道已经跑了多久
3. ✅ 停止时统计汇总：开始时间/结束时间/总时长/输出文件
4. ✅ 关闭二次确认：防止误操作丢失进度

**可以更好的地方**:
1. ⚠️ 缺少"正在识别中"的视觉反馈（比如闪烁图标）
2. ⚠️ 没有"复制整段文字"的快捷按钮
3. ⚠️ 缺少"打开输出文件夹"的快捷入口

**UI 设计方法论**:
> GUI 应该早介入（需求阶段就画草图），而不是功能完成后补。每实现一个核心功能，同步在界面上加对应控件，边开发边调 UI。

## 🔄 可复用模式库

### 模式 1: 大模型本地部署项目

**通用架构**:
```
音频捕获 → 预处理 (VAD/静音过滤) → 
深度学习模型 (FunASR/Qwen/etc) → 
后处理 (文本清洗) → 文件输出 + GUI 展示
```

**关键点**:
1. 虚拟环境隔离（避免依赖冲突）
2. 懒加载模式（点击开始再加载模型）
3. 进度反馈（加载中要有明确提示）
4. 降级方案（网络不好时的离线备用）

---

### 模式 2: GUI 控制后台进程

**核心逻辑**:
```python
class ControllerApp:
    def start(self):
        subprocess.Popen(cmd)  # 启动子进程
        
    def stop(self):
        process.terminate()     # 优雅终止
        
    def monitor(self):
        read_process_output()   # 读取 stdout/stderr
        update_gui_status()     # 实时更新界面
```

**关键点**:
1. 子进程独立运行，主线程负责界面
2. PID 文件标记进程状态
3. 退出前清理所有资源
4. 错误信息回传到 GUI 日志区
